# R Package Maintenance

Use this skill when the task touches package code, package metadata, exports, or package-level verification.

## Repo Triggers

Common triggers in this repo:
- `R/*.R`
- `DESCRIPTION`
- `NAMESPACE`
- `man/*.Rd`
- exported entrypoints `CPions()` and `CPquant()`
- package load or package check requests
- `tests/testthat` changes or focused test requests

## Repo Facts

- This repository is a single R package, not a monorepo.
- The main exported app entrypoints are `CPions()` in `R/CPions.R` and `CPquant()` in `R/CPquant.R`.
- Shared helpers live in `R/CPions_utils.R` and `R/CPquant_utils.R`.
- `NAMESPACE` and files under `man/` are generated by `roxygen2` and should not be edited by hand.
- `README.md` is generated from `README.Rmd`.
- The package has testthat tests under `tests/testthat`, including CPions snapshots/smoke tests and CPquant demo-data coverage.

## Working Rules

- Make the smallest correct change.
- Prefer editing existing functions over introducing extra abstraction unless reuse is clear.
- Preserve current package structure and naming unless the task explicitly requests a broader refactor.
- Avoid editing generated files directly when the source file is known.

## Generated File Rules

- If exports or roxygen docs change, edit roxygen comments in `R/*.R`, then regenerate docs.
- Do not hand-edit `NAMESPACE`.
- Do not hand-edit `man/*.Rd`.
- If README content changes, edit `README.Rmd`, then re-knit `README.md`.

## Verification

Use focused verification that matches the change.

Typical verification:
- `devtools::load_all()`
- `devtools::test()`
- `devtools::test(filter = "cpions-skyline")`
- `devtools::test(filter = "cpquant-demo-data")`
- `CPions()` for CPions changes
- `CPquant()` for CPquant changes

When documentation or metadata changed:
- `devtools::document()`
- build or check only if needed by the task

## Constraints

- Do not claim coverage beyond the existing `tests/testthat` checks that were actually run or inspected.
- Runtime assets under `inst/` are loaded via `system.file(...)`; keep those paths intact when changing app logic.
- Keep changes compatible with the package's current workflow unless the user explicitly asks to expand behavior.

## Completion Checklist

- Source files updated in `R/` or `README.Rmd` as needed
- Generated files regenerated only when required
- Relevant package load or app entrypoint verification attempted
- Final note states what was verified and what was not
